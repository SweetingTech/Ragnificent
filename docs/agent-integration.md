# Agent Integration Guide

This guide is for external LLM agents, tool-calling assistants, and automated pipelines that need to use RAGnificent as a retrieval-backed knowledge service.

The short version:

1. check `/health`
2. list corpora with `/api/corpora`
3. pick a corpus with `vector_count > 0`
4. query it with `/api/query`
5. use `answer` for the user-facing response and `hits` for citations / grounding

---

## Base URL

RAGnificent usually runs on:

```text
http://<host-or-ip>:8008
```

Examples:

```text
http://localhost:8008
http://192.168.1.50:8008
http://ragserver.local:8008
```

CORS is open, so browser-based agents and local-network clients can call the API directly.

---

## What An LLM Can Do

An external LLM can use this server to:

- discover which knowledge bases exist
- check whether a corpus is actually indexed
- create a new corpus pointed at a folder of files
- trigger ingestion or retry failed files
- query a specific corpus and get:
  - a synthesized answer
  - raw retrieved chunks
  - source filenames
  - section / chunk metadata

What matters operationally:

- RAGnificent does not decide which corpus your agent should use
- your agent should inspect corpus descriptions and choose the closest match
- `hits` are the authoritative grounding data
- `answer` is the convenience answer generated by the configured answer model

## Step 1: Check Service Health

```http
GET /health
```

Example response:

```json
{
  "status": "ok",
  "service": "ragnificent",
  "dependencies": {
    "qdrant": {
      "status": "ok",
      "url": "http://localhost:6333"
    },
    "embeddings": {
      "status": "ok",
      "provider": "openrouter",
      "url": "https://openrouter.ai/api/v1"
    },
    "answer": {
      "status": "ok",
      "provider": "openai",
      "url": "https://api.openai.com/v1"
    }
  }
}
```

Agent guidance:

- if `/health` is not `200`, treat the service as degraded
- if `qdrant` is not `ok`, retrieval is not reliable
- if `embeddings` is not `ok`, new queries may fail
- if `answer` is not `ok`, retrieval may still work, but answer generation may fail

## Step 2: Discover Available Corpora

```http
GET /api/corpora
```

Example response:

```json
[
  {
    "corpus_id": "smut",
    "description": "Erotic fiction knowledge base",
    "vector_count": 985,
    "query_endpoint": "/api/query"
  },
  {
    "corpus_id": "LLM",
    "description": "LLM and AI engineering books",
    "vector_count": 3578,
    "query_endpoint": "/api/query"
  }
]
```

Returned fields:

| Field | Meaning |
|---|---|
| `corpus_id` | The ID to pass into `/api/query` |
| `description` | Human-readable corpus summary |
| `vector_count` | Number of indexed chunks |
| `query_endpoint` | Usually `/api/query` |

Agent decision rule:

- ignore corpora with `vector_count == 0`
- choose the corpus whose `description` best matches user intent
- if nothing matches, create a new corpus and ingest before querying

---

## Step 3: Inspect One Corpus

```http
GET /api/corpora/{corpus_id}
```

Example:

```http
GET /api/corpora/LLM
```

This returns:

- the summary fields from `/api/corpora`
- the raw `config` block from that corpus’s `corpus.yaml`

That `config` block is useful because it tells an agent:

- the corpus description
- the source path stored in the corpus config
- the answer provider/model
- the embedding provider/model
- chunking strategy and token limits

Example shape:

```json
{
  "corpus_id": "LLM",
  "description": "LLM and AI engineering books",
  "vector_count": 3578,
  "query_endpoint": "/api/query",
  "config": {
    "models": {
      "answer": {
        "provider": "openai",
        "model": "gpt-5.4",
        "base_url": "https://api.openai.com/v1"
      },
      "embeddings": {
        "provider": "openrouter",
        "model": "qwen/qwen3-embedding-8b",
        "base_url": "https://openrouter.ai/api/v1"
      }
    },
    "chunking": {
      "default": {
        "strategy": "paragraph_sections",
        "max_tokens": 500,
        "overlap_tokens": 100
      }
    }
  }
}
```

## Step 4: Query A Corpus

```http
POST /api/query
Content-Type: application/json
```

Request body:

```json
{
  "query": "What's a good way to start erotic fiction?",
  "corpus_id": "smut",
  "top_k": 5
}
```

Fields:

| Field | Required | Meaning |
|---|---|---|
| `query` | yes | Natural-language question |
| `corpus_id` | strongly recommended | Which corpus to search |
| `top_k` | no | Number of chunks to retrieve, default `5` |
| `llm_model` | no | Optional answer-model override |

Important behavior:

- if you provide `corpus_id`, RAGnificent embeds the query using that corpus’s effective embedding config
- it retrieves chunks from that corpus’s Qdrant collection
- it then generates an answer from the retrieved context
- if you omit `corpus_id`, this is not multi-corpus retrieval; it becomes ad-hoc chat mode

Example response:

```json
{
  "query": "What's a good way to start erotic fiction?",
  "answer": "A good way to start erotic fiction is by setting the scene and building anticipation...",
  "hits": [
    {
      "id": "7c4087c2-7cca-5025-b3af-bb93e4eab1ee",
      "score": 0.63,
      "payload": {
        "file_name": "bestlesbianerotica2015.epub",
        "chunk_index": 11,
        "text": "Best Lesbian Erotica 2015 ...",
        "source": "D:\\Books\\smut\\bestlesbianerotica2015.epub",
        "corpus_id": "smut"
      }
    }
  ],
  "time": 1.61
}
```

Returned fields:

| Field | Meaning |
|---|---|
| `answer` | Model-generated response based on retrieved context |
| `hits` | Retrieved chunks |
| `hits[].score` | Similarity score |
| `hits[].payload.text` | Retrieved chunk text |
| `hits[].payload.file_name` | Source filename |
| `hits[].payload.chunk_index` | Chunk / section indicator |
| `time` | Query duration in seconds |

---

## How Another LLM Should Use The Response

If you are building an external LLM agent on top of RAGnificent, use the response like this:

### Minimal pattern

1. send the user’s question to `/api/query`
2. read `answer`
3. read `hits`
4. cite `hits[].payload.file_name`
5. if `hits` is empty, say the corpus did not return relevant material

### More reliable pattern

Treat `hits` as the primary truth source and `answer` as an assistant draft.

Good agent behavior:

- trust `hits[].payload.text` over the summarized `answer`
- use `answer` when it is coherent and aligned with the retrieved chunks
- if `answer` is missing or contains an error string, synthesize your own answer from `hits`
- cite the filenames from the retrieved chunks
- if the top hits are weak, ask a narrower follow-up question

Example orchestration logic:

```text
1. POST /api/query with {query, corpus_id, top_k}
2. If hits is empty:
   respond that no grounded material was found
3. If answer starts with "Error generating answer:":
   ignore answer and synthesize from hits
4. Otherwise:
   use answer, but keep citations from hits
```

This makes RAGnificent usable both as:

- a full RAG answerer
- a retrieval backend that feeds a separate orchestration model

## Step 5: Create A New Corpus

```http
POST /api/corpora
Content-Type: application/json
```

Example request:

```json
{
  "corpus_id": "my_docs",
  "description": "Internal policy documents",
  "source_path": "D:/company/policies",
  "llm_provider": "openai",
  "llm_model": "gpt-5.4",
  "embedding_preset": "pdf_general",
  "embedding_provider": "openrouter",
  "embedding_model": "qwen/qwen3-embedding-8b",
  "chunk_strategy": "paragraph_sections",
  "chunk_max_tokens": 700,
  "chunk_overlap_tokens": 100
}
```

Create-corpus fields:

| Field | Required | Meaning |
|---|---|---|
| `corpus_id` | yes | Alphanumeric plus `_` / `-`, max 64 chars |
| `description` | yes | Human-readable label |
| `source_path` | yes | Directory to ingest |
| `llm_provider` | no | Answer provider |
| `llm_model` | no | Answer model |
| `llm_base_url` | no | Answer endpoint override |
| `embedding_preset` | no | Preset ID such as `epub_general` or `pdf_general` |
| `embedding_provider` | no | Embedding provider |
| `embedding_model` | no | Embedding model |
| `embedding_base_url` | no | Embedding endpoint override |
| `chunk_strategy` | no | Chunking strategy |
| `chunk_max_tokens` | no | Chunk size |
| `chunk_overlap_tokens` | no | Overlap size |

Response:

```json
{
  "status": "created",
  "corpus_id": "my_docs",
  "message": "Corpus 'my_docs' created. Trigger ingestion with POST /api/ingest/run?corpus_id=my_docs",
  "query_endpoint": "/api/query"
}
```

After creation, you must ingest before the corpus can answer retrieval queries.

## Step 6: Ingest Or Retry Failed Files

Trigger ingestion:

```http
POST /api/ingest/run?corpus_id=my_docs
```

Retry only failed files:

```http
POST /api/ingest/run?corpus_id=my_docs&retry_failed_only=true
```

Important behavior:

- this endpoint is synchronous from the caller’s point of view
- large corpora can take a long time
- clients should use a generous timeout or poll `/api/ingest/status`

Current ingest status:

```http
GET /api/ingest/status
```

Example response:

```json
{
  "status": "running",
  "message": "Processing 5 of 9",
  "corpus_id": "LLM",
  "current_file": "D:\\Books\\LLM\\llmsinenterprise.pdf",
  "total_files": 9,
  "files_completed": 4,
  "files_processed": 0,
  "files_failed": 4,
  "percent_complete": 44.4,
  "retry_failed_only": true
}
```

## Model Overrides And Answer Generation

`llm_model` on `/api/query` is an optional override.

Current behavior:

- blank `llm_model` means: use the corpus’s configured answer model
- provided `llm_model` means: use that model through the current default answer provider from Settings

That means a model override is useful for ad-hoc experimentation, but it is not a full provider switch by itself.

Example:

```json
{
  "query": "Summarize the main idea",
  "corpus_id": "LLM",
  "llm_model": "gpt-5.4"
}
```

## Recommended Agent Workflow

```text
1. GET /health
   if status != 200, stop or degrade gracefully

2. GET /api/corpora
   choose a corpus with vector_count > 0

3. Optional: GET /api/corpora/{corpus_id}
   inspect config if provider/chunking details matter

4. POST /api/query
   body: { query, corpus_id, top_k: 5 }

5. Use:
   - answer for the draft response
   - hits for citations and validation

6. If vector_count == 0 or no corpus exists:
   POST /api/corpora
   POST /api/ingest/run?corpus_id=<id>

7. If ingestion partially fails:
   POST /api/ingest/run?corpus_id=<id>&retry_failed_only=true
```

---

## Notes For Agent Implementors

- no authentication is required in a typical local-network deployment
- do not expose this service to the public internet without adding auth
- `corpus_id` must be alphanumeric plus underscores / hyphens only
- `vector_count > 0` is the simplest signal that a corpus is queryable
- omitting `corpus_id` does not perform cross-corpus search; it falls back to chat mode
- if `answer` begins with `Error generating answer:`, your agent should still inspect `hits`
- large PDF corpora may take longer to ingest because embedding requests are remote and batched
- `GET /api/query/models` only reflects Ollama models on the configured host; it is not a universal model catalog for OpenAI, Anthropic, or OpenRouter
